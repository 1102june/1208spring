# 일상 개발 가이드 - 개발 중 서버 작업 방법

## 🎯 핵심 개념

**개발 중이어도 서버에 올릴 수 있습니다!** 

서버는 **완성된 기능**을 테스트하는 곳이고, **일상적인 개발**은 로컬에서 계속 진행합니다.

---

## 📋 개발 워크플로우

### ✅ 정상적인 개발 흐름

```
┌─────────────────────────────────────────┐
│  1. 로컬에서 개발                        │
│     - 코드 수정                          │
│     - 로컬 서버에서 테스트               │
│     - 안드로이드 앱도 로컬 서버 연결     │
└─────────────────────────────────────────┘
              ↓
┌─────────────────────────────────────────┐
│  2. 기능이 완성되면                      │
│     - 로컬에서 충분히 테스트             │
│     - 문제 없으면 서버에 배포            │
└─────────────────────────────────────────┘
              ↓
┌─────────────────────────────────────────┐
│  3. 서버에서 실환경 테스트               │
│     - 실제 서버에서 확인                 │
│     - 문제 발견 시 로컬로 돌아가서 수정  │
└─────────────────────────────────────────┘
              ↓
┌─────────────────────────────────────────┐
│  4. 계속 개발                            │
│     - 다시 로컬로 돌아와서 개발          │
│     - 다음 기능 완성 시 서버 배포 반복   │
└─────────────────────────────────────────┘
```

## 🔄 실제 작업 예시

### 시나리오: 북마크 기능 수정

#### 1단계: 로컬에서 개발

```kotlin
// NetworkModule.kt는 자동으로 로컬 서버를 사용
// (Debug 빌드 시 http://127.0.0.1:8080/ 자동 사용)
```

```bash
# 백엔드 로컬 서버 실행
cd C:\Users\subpa\IdeaProjects\WiseYoung_backend
.\gradlew bootRun

# 안드로이드 앱도 Debug 모드로 실행
# → 자동으로 로컬 서버(127.0.0.1:8080)에 연결됨
```

- 코드 수정
- 로컬에서 테스트
- 문제 있으면 수정 반복

#### 2단계: 기능 완성 후 서버 배포

```bash
# 1. 백엔드 빌드 및 업로드
cd C:\Users\subpa\IdeaProjects\WiseYoung_backend
.\빌드_스크립트.ps1
.\서버_접속_스크립트.ps1  # JAR 업로드

# 2. 서버에서 재시작
ssh root@210.104.76.139
sudo systemctl restart wiseyoung-backend

# 3. 안드로이드 앱은 Release 빌드로 테스트
# → 자동으로 서버(210.104.76.139:8080)에 연결됨
```

#### 3단계: 계속 개발

- 다시 로컬로 돌아와서 개발
- Debug 모드로 실행하면 자동으로 로컬 서버 연결

## 💡 핵심 포인트

### 1. 자동 환경 전환

안드로이드 앱은 이제 **자동으로** 환경에 맞는 서버를 사용합니다:

- **Debug 빌드** → 로컬 서버 (`http://127.0.0.1:8080/`)
- **Release 빌드** → 실제 서버 (`http://210.104.76.139:8080/`)

**수동으로 Base URL을 변경할 필요가 없습니다!**

### 2. 두 가지 모드로 작업

```
개발 중 (일상):
├─ 안드로이드: Debug 모드 실행
├─ 백엔드: 로컬 서버 실행
└─ 모두 로컬 환경에서 작업

배포 후 (테스트):
├─ 안드로이드: Release 빌드
├─ 백엔드: 서버에 배포
└─ 실제 서버 환경에서 테스트
```

## 🔧 빈번한 질문

### Q: 매번 서버에 올려야 하나요?
**A**: 아니요. 완성된 기능만 서버에 올리세요. 일상 개발은 로컬에서.

### Q: 서버에 올렸는데 문제가 발견되면?
**A**: 로컬로 돌아가서 수정 → 다시 서버 배포.

### Q: 여러 사람이 동시에 작업하면?
**A**: Git을 사용하여 버전 관리. 각자 로컬에서 개발 → 서버는 공유.

### Q: 서버 데이터와 로컬 데이터는?
**A**: 서로 다릅니다. 개발 중에는 로컬 DB 사용, 배포 후에는 서버 DB 사용.

## 🚀 권장 배포 주기

### 작은 기능 단위로 배포 (권장)

✅ **좋은 예**:
- 월요일: 북마크 기능 완성 → 서버 배포
- 화요일~목요일: 로컬에서 캘린더 기능 개발
- 금요일: 캘린더 기능 완성 → 서버 배포

❌ **나쁜 예**:
- 2주 동안 모든 기능 개발 → 한 번에 배포
  → 문제 발생 시 원인 파악 어려움

## 💻 실전 예시

### 예시 1: 평일 개발

```bash
# 월요일 아침
1. 로컬 백엔드 서버 실행
2. 안드로이드 앱 Debug 모드 실행
3. 북마크 기능 개발 시작

# 월요일 오후
4. 기능 완성, 로컬에서 테스트 완료
5. 서버에 배포
6. Release 빌드로 서버 테스트 → 성공!

# 화요일 아침
7. 다시 로컬 환경으로 돌아옴
8. 다음 기능 개발 시작
```

### 예시 2: 서버에서만 발생하는 문제

```bash
# 서버 배포 후 에러 발생
1. 서버 로그 확인: sudo journalctl -u wiseyoung-backend -f
2. 문제 파악: "데이터베이스 연결 오류"

# 로컬로 돌아와서 수정
3. 로컬에서 동일한 환경 재현 시도
4. 문제 수정
5. 로컬에서 테스트
6. 다시 서버 배포
```

## 🎯 요약

1. **일상 개발**: 로컬 환경 (127.0.0.1)
2. **기능 완성**: 서버 배포 (210.104.76.139)
3. **자동 전환**: Debug/Release 모드에 따라 자동
4. **계속 개발**: 배포 후 다시 로컬로 돌아와서 개발

**서버는 "완성된 버전을 테스트하는 곳"**, **로컬은 "개발하는 곳"**입니다!

## 📞 문제 발생 시

### 로컬 개발 중 문제
- 로컬 서버 로그 확인
- 안드로이드 Logcat 확인
- 코드 수정 후 재실행

### 서버 배포 후 문제
- 서버 로그 확인: `sudo journalctl -u wiseyoung-backend -f`
- 문제 파악 후 로컬에서 수정
- 다시 배포
